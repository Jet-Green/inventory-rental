## Phase 1 — ручная проверка (без реальных интеграций)

### Предпосылки
- Backend запущен на `http://localhost:4000` (prefix `/api`).
- Frontend запущен на `http://localhost:3068`.
- В `.env` ничего не меняем.

### 1) Публичные страницы и CTA
- Открыть `/`:
  - Видна hero-секция.
  - Кнопки: **«Перейти в каталог»** ведёт на `/listings`.
  - Кнопка **«Выложить своё объявление»** ведёт на `/cabinet/select`.
- В верхнем меню (layout `default`) на любой публичной странице:
  - Пункт **«Каталог»** ведёт на `/listings`.
  - CTA **«Выложить своё объявление»** ведёт на `/cabinet/select`.

### 2) Каталог (новый маршрут)
- Открыть `/listings`:
  - Загружается каталог (карточки).
  - Фильтры отображаются.
  - Нажать **«Взять»** на карточке или клик по карточке → переход на `/listings/:id`.
- Проверить, что старый маршрут `/rental-items` всё ещё работает (обратная совместимость).

### 3) Карточка объявления (новый маршрут)
- Открыть `/listings/:id` (из каталога):
  - Рендерится заголовок/описание/картинка.
  - Рендерится календарь (как раньше).
  - Форма бронирования доступна только при роли renter/admin:
    - если не авторизован — при попытке отправки редирект на `/auth`.

### 4) Новый API фасад (read-only)
- `GET /api/categories`:
  - Возвращает `{ data: [...] }`, элементы содержат `id`, `title`, `slug`, `isActive`.
- `GET /api/listings?page=1&limit=12`:
  - Возвращает `{ data, page, total, totalPages }`.
  - В `data` нет PII владельца.
- `GET /api/listings/:id`:
  - Возвращает `{ listing }`.

### 5) Админка и guards
- Зайти в `/admin` без роли admin:
  - Front middleware должен редиректить в `/cabinet` (как и раньше).
- Дёрнуть `GET /api/admin/dashboard` без admin cookie:
  - Должно быть `401` (если не авторизован) или `403` (если не admin).
